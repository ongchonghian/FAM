## Design Context

### Users
Technical architects, bank CTO/CIO offices, Singapore government regulators (MAS, IMDA), and enterprise stakeholders evaluating federated analytics infrastructure. Viewed in office settings on large monitors — decision-makers who need to understand cryptographic data flows without seeing raw code.

### Brand Personality
**Authoritative · Cryptographic · Precise** — the interface should feel like reading certified technical documentation for regulated financial infrastructure, not a startup pitch deck. Trust signals matter more than delight.

### Aesthetic Direction
Dark mode is correct and intentional: "secure distributed infrastructure" viewed in institutional tech environments. Deep navy backgrounds (`#0f1626`, `#0d1526`), with controlled neon accents for active/live state — green for PET/analytics (ASEs), cyan for data paths, gold for query flows, purple/pink for consumer tiers. The palette is thematic: each color maps to a structural role in the mesh.

Anti-references: avoid SaaS marketing aesthetics, startup glassmorphism, casual rounded-everything. The product must feel like infrastructure, not a demo.

### Design Principles
1. **Flow clarity over decoration** — every color and glow exists to convey which components are active in a given protocol step, not for aesthetic effect
2. **Dim to focus** — inactive nodes at 35% opacity create contrast that directs attention; this is the primary navigation metaphor of the InAction page
3. **Structural integrity** — band backgrounds (PEPPOL, dex ecosystems) provide spatial orientation and must stay legible regardless of active state
4. **Technical vocabulary** — labels use exact protocol names (AS4, PINT-SG, KKRT-PSI, SD-JWT-VC); don't simplify
5. **Information density is intentional** — this is not a consumer UI; experts expect rich metadata per node
