01 — DESIGN PRINCIPLES

Six Invariants the Architecture Must Preserve

The mesh is not a single product but a set of cooperating components governed by six invariants. Every flow, every protocol choice, every governance decision can be traced back to one of these.

INVARIANT 1
Data Locality
Invoice payloads remain at the data owner (C1, C4) or at an accredited dex-onboarded Data Provider that holds them on the owner's behalf — currently the Invoice Repository (IR) for PEPPOL invoices, forwarded by C2 in parallel to the standard 4-corner path. ASEs hold only derivative cryptographic material: hashes, commitments, attestations, gradients, indexes. No third-party operator holds an unmediated queryable copy of an invoice.
INVARIANT 2
Computation-to-Data
Queries travel to the data, not the reverse. The Query Broker decomposes an analytical request into sub-queries that execute at the relevant ASEs. Only results — typically aggregates, proofs or boolean answers — return to the consumer.
INVARIANT 3
Owner-Mediated Disclosure
Suppliers (C1) and buyers (C4) are the data owners. Every disclosure to a third party is mediated either by an owner-held verifiable credential or by an explicit, revocable, attribute-bound consent grant. Operators have delegated custody, never ownership.
INVARIANT 4
Universal Auditability
Every analytical event — issuance, presentation, query, federation, training round — is recorded in a tamper-evident transparency log with cryptographic inclusion proofs. Data owners and independent auditors can verify every event without trusting any operator.
INVARIANT 5
PEPPOL-Native Routing
All discovery reuses the SML/SMP DNS-based discovery model. Analytical services are described by URN-coded capability tags in a sibling registry (the Analytical Service Registry — ASR). No parallel routing fabric exists, no SML hard-fork.
INVARIANT 6
Defence in Depth
No single PET is load-bearing. The mesh composes verifiable credentials, federated learning, secure enclaves, MPC, PSI and ZKPs as peers. If any single primitive degrades — a TEE side-channel, a ZKP soundness flaw — the surrounding layers preserve confidentiality and integrity.
02 — COMPONENT ANATOMY

Inside Each Architectural Component

Seven logical components constitute the mesh. Each is described by its location, purpose, key interfaces, internal data, and the operator who runs it.

COMPONENT 1 · ASE
Analytical Service Endpoint
co-located with each Access Point · one ASE per AP
The atomic unit of the mesh. An ASE is a logically distinct module operated by the same legal entity as the AP, holding only derivative cryptographic material about the invoices that AP has processed. It exposes capabilities — VC issuance, ZKP attestation, FL local-training, PSI/MPC participation, TEE-attested execution — discoverable through the ASR.
Operated byIMDA-accredited Access Point provider (under a new Analytical Services Annex to the existing AP accreditation)
Data heldInvoice content hashes (SHA-256) · Pedersen commitments of values · signed attribute attestations · VC issuance keys · FL local-model parameters · audit log entries · ABAC policy bundles
Data not heldInvoice payloads · line items · personal data of buyers/suppliers beyond UEN-level identifiers
Interfaces/issue · /present · /query · /psi · /fl-round · /attest · /log
RuntimeConfidential VM (Intel TDX, AMD SEV-SNP, AWS Nitro Enclaves) with remote attestation
Failure modeIf an ASE is offline, queries that span its data are answered with a marked degradation flag; cross-AP queries proceed with reduced cohort
COMPONENT 2 · QB
Query Broker
federated · one per ASE operator · peer-to-peer between brokers
The orchestration layer. A consumer's analytical request is a query plan, not a raw SQL string: the broker translates intent into a federated execution plan, performs ABAC + VC verification, dispatches sub-queries, aggregates results, writes audit entries and returns the final response.
Operated bySame operator as the ASE (federated topology — brokers communicate peer-to-peer using a defined protocol)
InputsQuery intent (URN-coded capability) · consumer's VC presentation · execution-time parameters · privacy budget allocation
OutputsResult + result-attestation (signed) · privacy-budget receipt · transparency-log inclusion proof
SecurityGNAP-mediated authentication (RFC 9635) · ABAC policy engine · rate limits per consumer credential
Latency targetp50 < 200 ms for VC verification queries; p50 < 2 s for single-ASE ZKP queries; p50 < 30 s for multi-party PSI
COMPONENT 3 · VCIS
Verifiable Credential Issuer Service
a function of each ASE · separately keyed
Issues SD-JWT-VC or BBS+-signed credentials over invoice attributes selected by C1 or C4. The issuer-public-key chain anchors to the IMDA AP accreditation root, allowing any verifier to validate issuance offline. Credentials are revocable through a status-list mechanism (RFC draft draft-ietf-oauth-status-list).
Credential formatSD-JWT-VC (default) · W3C VCDM 2.0 with bbs-2023 for BBS+ · ISO mdoc on request
Trust anchorIMDA-issued X.509 cert chain · DID:web for the ASE operator
RevocationOAuth 2.0 status list bitstring · published every 60 s
Holder bindingCryptographic key bound to C1's or C4's PEPPOL identity (UEN + PEPPOL Participant ID)
Selective disclosureSD-JWT salted hashes for plain attributes · BBS+ for unlinkable presentations
COMPONENT 4 · TLOG
Transparency Log
one per ASE · DLT-anchored Merkle roots
An append-only Merkle tree (Certificate Transparency design — RFC 6962) recording every analytical event. Periodic Signed Tree Heads are anchored to a public DLT (XDC, Polygon or Ethereum mainnet, agnostic at the protocol layer), creating an immutable global witness. Data owners can fetch inclusion proofs on demand.
Logged eventsVC issuance · VC presentation · query receipt · query result hash · FL round commit · revocation event
Proofs offeredInclusion (RFC 6962 §2.1.1) · consistency (§2.1.2) · per-data-owner derivation
AnchoringHourly Signed Tree Heads anchored to public DLT · separate DLT layer is opaque to PEPPOL operations
VerificationAny third party with the public log key + DLT chain root can verify any logged event without operator cooperation
COMPONENT 5 · ASR
Analytical Service Registry
sibling of SMP · accessed via SML DNS root
The discovery extension. For each PEPPOL participant, the ASR enumerates the analytical capabilities its serving AP supports. Capability identifiers follow a URN scheme so consumers can discover services without knowing which AP serves a given supplier.
URN schemeurn:peppol:analytics:<capability>:<version>
DNS patternB-{md5(participant)}.iso6523-actorid-upis.{sml-zone} with new _analytics._tcp SRV records
Examplesduplicate-check:1 · fedlearn:fraud:1 · zkp:gst-correctness:1 · range-proof:receivables:1
GovernanceOpenPeppol catalogue under a new Analytical Services Profile
COMPONENT 6 · PE
Policy Engine
embedded in Query Broker · ABAC + GNAP
Evaluates whether a consumer's request — given its presented VCs, the data-owner's standing consents, and the regulatory context — should be admitted. Policy is expressed declaratively; decisions are signed and themselves logged.
Policy languageRego (OPA) · Cedar · or XACML 3.0 — implementation choice
InputsSubject attributes (consumer VCs) · resource attributes (invoice class, sector) · action (capability URN) · context (time, jurisdiction)
OutputsPermit / Deny / Indeterminate · obligations (e.g. DP ε budget) · advice (e.g. require additional VC)
Standard policiesMAS sectoral codes · IRAS audit codes · PDPC data-class codes · data-owner standing consents
COMPONENT 7 · ASE-IR

03 — THE SEVEN CORE FLOWS

Sequence Diagrams for Every Mesh Operation

Seven flows cover all mesh operations. Each is depicted as a sequence diagram with explicit actors, messages and side-effects. The original 4-corner AS4 flow remains untouched. F0 is the new IR ingestion path; F6 is the multi-Provider fan-out pattern that closes the loop on cross-Provider analytics. Jump to: F0 · F1 · F2 · F3 · F4 · F5 · F6.

F0 PEPPOL → IR Ingestion — invoice payload forwarded to the Invoice Repository in parallel to the 4-corner path
parallel to AS4 send · IR is dex-onboarded Data Provider
Side-effects: IR persists the full PEPPOL invoice payload (the only Provider in the mesh that does so). ASE-IR persists ~1 KB of derivative material per invoice. C1 receives an SD-JWT-VC issued by IR's VCIS and bound to its holder key. The transparency log gains one signed entry. The standard 4-corner C2→C3→C4 path is unchanged and runs concurrently. Latency budget for the IR leg: <700 ms additional to standard AS4 forward, asynchronous to the buyer-side delivery.

pitstop-core does not appear as a lifeline. The C2 → IR forwarding is implemented as a post-transmission callback on the AP itself — a new configuration item under the Analytical Services Annex — and does not route through pitstop-core's /api/v1/data/forward route. This keeps IR's ingestion path independent of the dex exchange plane and avoids coupling IR's availability to the per-Org pitstop instance. [TODO: verify against IR Jira; if IR has chosen to implement ingestion on top of pitstop-core's /store route, the C2 column gains a pitstop-core lifeline label.]
F1 Issuance Flow — derivative material is created when an invoice transmits
synchronous with AS4 send
Side-effects: ASE persists ~1 KB of derivative material per invoice. C1 receives a VC bound to its holder key. Transparency Log gains one signed entry. The invoice itself never leaves the AS4 path. Latency budget: <500 ms additional to standard AS4 forward.
F2 Consent & Disclosure Flow — data owner releases a VC to a third party
offline-verifiable · zero broker contact
Key property: the VCIS does not learn that this presentation occurred. The holder controls timing and selection. Banking secrecy implications under Section 47 of the Singapore Banking Act are minimised because no AP or operator is in the disclosure path.

Consent Manifest pre-condition: for VPs that target a Data Provider's ASE (e.g. ASE-IR for invoice-domain claims), the data owner has previously signed a Consent Manifest via admin-ui's Consent Manifest Manager (holder-key signing in the browser session). The signed manifest is registered in admin-corev2 and distributed to ASE-IR; both hold a copy and verify it independently. At query time the FAM Query Broker enforces a two-gate sequential check — Gate 1 at admin-corev2 (eligibility, via Policy Engine) and Gate 2 at ASE-IR (execution integrity, against its local copy). See F3 for the full sequence. Revocation propagates via Status List 2021 within ≤ 60 s to ASE-IR. [TODO: requires admin-corev2 schema extension — see §02b admin-corev2 card]
F3 Single-ASE Query Flow — sequential manifest verification, then PET execution
target latency p50 < 2 s · two enforcement gates
Two enforcement gates, sequential. Gate 1 (admin-corev2 via PE) is the eligibility check: does a valid, non-revoked Consent Manifest exist for this consumer-owner-element triple, with budget remaining and the requested query class permitted? If DENY, the flow terminates at step 6 — ASE-IR is never contacted, no ε is consumed, and the denial is logged to TLog. Gate 2 (ASE-IR's local manifest copy) is the execution-integrity check: do the sub-query parameters the QB has dispatched match the manifest copy ASE-IR holds locally? This defends against a compromised QB that has passed Gate 1 but is now dispatching a malformed or over-scoped sub-query. If the two manifest copies diverge, ASE-IR returns a signed refusal and the QB logs a Tier 2 incident (manifest drift between the platform registry and the Provider's local copy) — IMDA notification within 2 hours.

Why sequential, not concurrent. Concurrent execution opens a window where ASE-IR is partway through a TEE workload when admin-corev2 returns a denial. Cleaning up partially executed PET workloads (rolling back a DP-noised result, refunding committed ε, sealing TEE state) is operationally complex and introduces race conditions in the ε ledger. The latency cost of one admin-corev2 round-trip is small relative to ZKP / PSI execution.

Revocation propagation SLA: if the data owner revokes a manifest after Gate 1 passes but before ASE-IR's Gate 2, ASE-IR's local copy must reflect the revocation within ≤ 60 seconds (matching the TLog STH cadence). [TODO: define revocation propagation SLA in ASA §9]
F4 Cross-ASE Federation Flow — PSI / MPC across multiple corners
duplicate-finance · sanctions · aggregate exposure
What leaks: only the cardinality of the intersection (typically 0 or 1 for duplicate-finance). What does not leak: any ASE's full invoice set, party identities of unaffected suppliers, query history of other consumers. The protocol can be hardened with threshold MPC (3-of-5 ASEs) against collusion.
F5 Federated Learning Round — model trains across all corners
scheduled · DP-noised · secure aggregation
Privacy guarantee: the aggregator cannot recover any individual gradient because of the secure-aggregation masking protocol. DP noise added at each ASE bounds membership inference. Reference: J.P. Morgan / NVIDIA FLARE production deployment patterns.
F6 Multi-Provider Fan-Out — one consumer query spans multiple Data Providers
parallel sub-queries · independent Consent Manifest verification · independent log entries
Properties: each Data Provider verifies its own Consent Manifest independently — neither the consumer, nor the QB, nor the other Provider can substitute authority for it. Each Provider's transparency log records its own sub-query event; there is no global log entry. The QB returns only the assembled result; the Providers never learn of each other. Status: not implemented in dex today — pitstop-core executes single-Provider REST queries via getMessages. 04 — WORKED EXAMPLES

Three Scenarios End-to-End

How the abstract architecture behaves under real-world demands. Each example pins specific PETs, components and flows to a concrete business scenario.

EXAMPLE A · INVOICE FINANCING
Acme Pte Ltd seeks S$200K working capital from DBS
Acme has 47 unpaid invoices to several large buyers. DBS needs to verify total receivables exceed S$200K, no invoice has been pledged elsewhere, and counterparty payment behaviour is acceptable — all without seeing line items.
  • C1 (Acme) Holds 47 SD-JWT-VCs issued by its AP — one per invoice — bound to its UEN-anchored holder key. Flow F1 already executed at issuance time
  • C5 (DBS) Issues a presentation request: "prove Σ(unpaid totals) > 200,000 SGD over invoices issued in the last 90 days, and prove non-membership of each invoice ID in any other bank's pledged-invoice accumulator." Flow F2 + ZKP capability urn:peppol:analytics:range-proof:receivables:1
  • C1 (Acme) Generates a Bulletproof range-proof over the Pedersen commitments embedded in each VC, plus 47 accumulator non-membership proofs against the cross-bank duplicate-finance accumulator. Bulletproofs · transparent setup · ~700 ms total prove time
  • DBS Verifies all proofs offline against IMDA-anchored AP issuer keys. No raw invoice data, no line items, no buyer identities (beyond UEN-class category) are revealed. Verification <50 ms
  • Broker (DBS-side) Logs the financing decision in DBS's transparency log. Acme can later prove what was disclosed — protective if DBS is ever audited or breached. Append-only Merkle entry · DLT-anchored hourly
Outcome: Financing decision in seconds. DBS sees only "Σ > 200K" and "no duplicates"; Acme retains complete confidentiality over commercial terms with its buyers. Section 47 banking-secrecy concerns minimised.
EXAMPLE B · DUPLICATE-FINANCE DETECTION
Bank X presents invoice INV-12345 for financing — has any other bank already taken it?
In Singapore today, the MonetaGo / ABS Trade Finance Registry handles this with hashed fingerprints. The mesh upgrades the protocol with formal PSI guarantees: the registry operator does not learn the queries.
  • Bank X Computes h = SHA-256(canonicalised invoice fields) and asks "is h in any other bank's pledged-invoice set?" capability urn:peppol:analytics:duplicate-check:1
  • Query Broker Discovers all participating bank ASEs via the ASR. Initiates an N-party PSI protocol (Bank X as client, all other banks' ASEs as servers). Flow F4 · KKRT-style PSI · OPRF
  • All bank ASEs Engage in oblivious PRF rounds. Each bank's pledged-invoice set is encoded as a Bloom filter of OPRF-evaluated hashes; intersection is computed obliviously. ~6 round trips · <30 s wall-clock for cohort of 12 banks
  • Broker Returns binary result {0, 1} to Bank X. Each ASE that contributed signs the federation-event log entry — providing irrefutable evidence the protocol ran correctly. Multi-party signature · TLog inclusion proof
Outcome: Bank X learns whether the invoice is already pledged. What no one learns: which other banks have which invoices, what other queries Bank X is making, whether unrelated invoices are pledged. Strict improvement over hashed-fingerprint registries.
EXAMPLE C · IRAS GST AUDIT
IRAS investigates GST anomalies in the F&B sector for FY2026
IRAS already receives a copy of every in-scope invoice under the GST InvoiceNow Requirement. The mesh adds a privacy-preserving analytical layer for queries that span multiple taxpayers without de-anonymising any single one.
  • IRAS analyst Submits a query: "compute the distribution of GST-to-revenue ratios across F&B SSIC code 56 firms with annual turnover S$1m–10m, with DP ε ≤ 1." capability urn:peppol:analytics:dp-aggregate:1
  • Policy Engine Verifies IRAS's audit-purpose VC, checks the requested ε against IRAS's annual privacy budget, returns a permit + obligation attaching ε = 1. Cedar policy · MAS / IRAS sectoral codes
  • Broker Dispatches sub-queries to all ASEs serving SSIC-56 firms in scope. Each ASE computes the local statistic, adds calibrated Laplace noise, returns the noisy result. Sensitivity Δ bounded by per-firm clipping · σ = Δ / ε
  • Broker Aggregates per-ASE noisy results. Returns histogram to IRAS. Logs ε spent (1) against IRAS's annual budget (e.g. 50 / year). Composition theorem applied · receipt issued
  • Data owners Each affected firm can fetch from the transparency log the fact that an IRAS audit query touched its data, and the ε spent. Cannot determine which other firms were in the cohort. Inclusion proof · per-owner derivation
Outcome: IRAS gains valuable sectoral signal without any single firm's GST ratio being learnable from the result. The architecture extends IRAS's existing 5th-corner role to richer analytics without requiring new data flows.

06 — ASR DISCOVERY INTERNALS

How a Consumer Finds the Right Endpoint

The Analytical Service Registry reuses PEPPOL's existing DNS-based discovery model. No new global registry, no central directory, no SML hard-fork.

URN scheme for analytical capabilities

Capabilities follow a hierarchical URN namespace. Versions allow non-breaking extension; obsolete capabilities can be deprecated without disturbing live consumers.

# pattern
urn:peppol:analytics:<capability>:<version>

# examples
urn:peppol:analytics:duplicate-check:1
urn:peppol:analytics:range-proof:receivables:1
urn:peppol:analytics:zkp:gst-correctness:1
urn:peppol:analytics:fedlearn:fraud:2
urn:peppol:analytics:dp-aggregate:1
urn:peppol:analytics:psi:sanctions:1

Governance & versioning

OpenPeppol's Coordinating Community pattern, already used for the billing post-award workflow, is reused for analytical capabilities. A new sub-community — the Analytical Services Coordinating Community — owns the URN registry.

  • Major version: incompatible change · breaking
  • Minor version: additive · backwards-compatible
  • Deprecation: 12-month sunset window with parallel availability

08 — FAILURE MODES & RESILIENCE

How the Mesh Degrades Gracefully

Each component can fail. The mesh's graceful-degradation property is that no single failure cascades into a confidentiality violation or a data loss.

!
Single ASE offline
Queries that span the offline ASE return a marked degradation flag and a reduced cohort. Document path (AS4) unaffected. Recovery: automatic on ASE restart.
!
Query Broker outage
Consumers fail over to peer brokers via ASR-published replicas. F2 disclosure flow continues unaffected. Recovery: peer-to-peer broker topology eliminates single-point dependency.
!
Transparency log inconsistency
Mismatch between Signed Tree Head and DLT anchor triggers automatic gossip-based reconciliation. Consumers detecting inconsistency proofs raise public alerts. Recovery: all events are deterministic; replay rebuilds the log.
!
DLT anchor chain forks / outage
Anchoring is best-effort and chain-agnostic. The mesh tolerates DLT outages; Signed Tree Heads queue and anchor on chain recovery. Multi-chain anchoring optional for highest assurance. Recovery: chain-agnostic by design.
!
VCIS key compromise
All VCs from the affected key window are revoked via the status list. Re-issuance happens against a fresh key. Transparency log allows post-hoc audit of forged-VC scope. Recovery: 60-second status-list refresh window limits exposure.
!
Privacy-budget exhaustion
A consumer that has exhausted its annual ε is rejected by the policy engine. Data owners can review the consumer's complete query history from their derived transparency-log view. Recovery: budget renews on policy-engine renewal cycle.
!
SML / SMP outage
Inherited PEPPOL behaviour: cached SMP records continue to function during transient outages. ASR shares this caching behaviour. Recovery: standard PEPPOL operational practice applies.
!
Federated-learning round failure
If < minimum quorum of ASEs participate in a round, the round is aborted; no model update committed. Aggregator publishes the failure to the transparency log. Recovery: skip round, retry next scheduled cycle.
The deepest resilience property: because no component has unique custody of any invoice, no failure of any operator can result in invoice loss. The worst case for the mesh is analytical unavailability, never data unavailability. The 4-corner document path is independently resilient and unaffected.
